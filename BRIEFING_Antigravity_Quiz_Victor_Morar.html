01
Context — Ce există, ce se construiește
Principiul de bază

Nu reconstruim nimic de la zero. Codul existent (quiz-files.zip) conține o infrastructură React + TypeScript completă și funcțională. Sarcina este să înlocuim conținutul (tipologii, întrebări, texte, scoring) și să adaptăm logica de clasificare de la 3 axe la 6 tipologii prin scoring acumulat.

Codul existent a fost construit pentru un alt proiect (educație online pentru femei, expert Daniela Ionaș, 3 profiluri: BEG/EXP/NEW). Arhitectura tehnică este solidă și reutilizabilă complet.

Quiz-ul lui Victor funcționează pe o logică diferită: nu există o primă întrebare care clasifică omul — clasificarea rezultă din scorul acumulat pe parcursul tuturor întrebărilor. La final, tipologia cu cele mai multe puncte câștigă.

03
Cele 6 Tipologii — Definiții Complete
Principiu

Tipologiile sunt bazate pe problema structurală, nu pe starea emoțională. Fiecare antreprenor se regăsește în UNA ca problemă principală. Fiecare tipologie are o culoare de identificare folosită consistent în tot codul și UI.

R · Recrutare
Nu găsesc oameni buni
Angajează în panică, ia primul care apare. Cercul vicios: angajează → nu merge → concediază → angajează. Crede că problema e piața muncii.
Curs → Etapa 2 + Etapa 3
"Nu oamenii sunt problema. E lipsa procesului de recrutare."
Re · Retenție
Formez oameni care poi pleacă
Investește luni în formare. Fix când omul devine productiv, pleacă la concurență. Know-how-ul e în capul oamenilor, nu în procese documentate.
Curs → Etapa 3 + Etapa 4
"Dacă expertiza e doar în capul oamenilor, normal că pleacă cu ea."
M · Motivație
Plătesc dar nu primesc rezultat
Angajații vin la program, nu la performanță. Salariu fix fără KPI. Profitul per angajat scade cu fiecare om nou. Bonusurile nu schimbă nimic.
Curs → Etapa 4
"Salariul fix fără KPI omoară responsabilitatea."
D · Delegare
Fac eu totul, că altfel iese prost
El ESTE businessul. 12+ ore/zi, telefonul mereu pornit. A încercat să delege — tot el a refăcut. Cea mai frecventă tipologie din audiența lui Victor.
Curs → Etapa 1 + Etapa 5
"Le-ai dat sarcini fără să le dai un SISTEM."
C · Control
Nu știu ce se întâmplă în firma mea
Fără cifre, fără raportare, decizii pe instinct. Află că ceva nu merge doar când e deja dezastru. Pierde bani fără să știe unde.
Curs → Etapa 4
"Dacă nu măsori, nu poți îmbunătăți."
S · Structură
Cresc, dar totul se complică
Cifra urcă, echipa crește, dar fiecare manager face cum vrea. Nu există organigramă reală, nu există procese standard. Haosul crește odată cu firma.
Curs → Etapa 2 + 3 + 6
"Nu poți scala haosul. Întâi faci SISTEM, apoi crești."

04
Logica de Clasificare — Cum Funcționează Scoring-ul

Fiecare răspuns adaugă puncte la unul sau mai multe contoare de tipologie. La final, classification.ts citește contoarele și returnează tipologia cu scorul maxim ca primară și a doua ca secundară.

// types.ts — modificări necesare

export type ProfileAxis = 
  'R'   // Recrutare
  | 'Re'  // Retenție
  | 'M'   // Motivație
  | 'D'   // Delegare
  | 'C'   // Control
  | 'S';  // Structură

// În QuizState — câmp nou de adăugat
scores: Record<ProfileAxis, number>;

// Inițializare în createInitialState()
scores: { R: 0, Re: 0, M: 0, D: 0, C: 0, S: 0 }
// classification.ts — logica nouă completă

export function performClassification(state: QuizState): QuizState {
  const scores = state.scores;
  
  // Găsește tipologia primară (max score)
  const primary = (Object.keys(scores) as ProfileAxis[])
    .reduce((a, b) => scores[a] >= scores[b] ? a : b);
  
  // Găsește tipologia secundară (al doilea max)
  const secondary = (Object.keys(scores) as ProfileAxis[])
    .filter(k => k !== primary)
    .reduce((a, b) => scores[a] >= scores[b] ? a : b);

  return {
    ...state,
    phase: 'PROFILE_REVEAL',
    profileAxis: primary,
    secondaryProfile: secondary,
    resolvedProfile: primary,
  };
}

// TIE-BREAKER: dacă 2 tipologii au scor egal,
// Q6 (butonul magic) decide — are weight x2.
// Aplică weight la adăugarea scorului din Q6:
// scores[code] += points * 2
Cum se adaugă punctele în machine.ts

Fiecare event de tip ANSWER_Qx primește în payload un array de scoruri: scores: Partial<Record<ProfileAxis, number>>. Reducer-ul face merge(state.scores, payload.scores) — adună punctele la contoarele existente. Nu înlocuiește, adună.

Ecranul de Rezultat — Anatomie

Ecranul de rezultat (E16) afișează diagnoza personalizată. Conține întotdeauna tipologia primară (scor max) și tipologia secundară (al doilea scor). Nu afișează scorurile numerice — afișează descrierea și soluția.

Antet
[PRENUME], problema ta principală este:
Denumirea tipologiei primare + badge culoare
Problema primară
Descriere scurtă a durerii — 2–3 rânduri. Exact ce simte el. Efectul: „parcă vorbește despre mine."
Problema secundară
„Ai și o problemă secundară: [tipologia 2]" — scurtă mențiune. Nu intră în detalii.
VSL Primar
Video embed sau link — VSL-ul corespunzător tipologiei primare. Trimis manual de consultant prin WhatsApp.
VSL Secundar
Dacă tipologia secundară e diferită de primară — al doilea VSL opțional.
CTA
„Înscrie-te pe lista de așteptare" → link lista de așteptare. Variante text pe temperatură HOT/WARM/COLD.
Social Proof
Screenshot sau citat de la un client real al lui Victor. Relevant pentru tipologia identificată.
Fluxul după rezultat

Omul e redirecționat pe WhatsApp. Consultantul primește datele (nume, email, telefon, Instagram, tipologie primară) și îi trimite manual VSL-ul. Nu se trimite VSL automat. Consultantul continuă vânzarea pe DM conform strategiei DM existente.

09
Reguli de Implementare
1
Scoring acumulat, nu ramificat. Nu există o primă întrebare care clasifică omul. Toate cele 6 întrebări cu scoring rulează pentru toți utilizatorii în aceeași ordine. Clasificarea se face o singură dată, la final, în classification.ts.
2
Q6 = tie-breaker cu weight x2. Dacă după Q1–Q5 două tipologii au scor egal sau apropiat, răspunsul de la Q6 (butonul magic) decide. Implementare: adaugă punctele din Q6 de două ori, sau multiplică cu 2 în reducer.
3
Rezultat primar + secundar. Ecranul de rezultat afișează întotdeauna DOUĂ tipologii: cea cu scor maxim (primară) și a doua ca scor (secundară). QuizState trebuie să stocheze ambele: profileAxis (primar) și secondaryProfile (secundar).
4
Datele se colectează progresiv. Prenume → E2. Instagram → E5 (după 2 întrebări). Email → E8 (după 4 întrebări). Telefon → E11 (după 6 întrebări). Ordinea nu se schimbă. Omul nu simte că i se cer date — simte că primește ceva în schimb la fiecare pas.
5
Sandler Pain nu acumulează puncte. Ecranele E12, E13, E14 sunt exclusiv pentru adâncirea durerii și crearea urgenței. Nu influențează clasificarea. Nu au scoring.
6
